ما هي Broken Access Control؟
What is Broken Access Control?

Broken Access Control (BAC) هي ثغرة تحدث عندما يتمكن مستخدم من الوصول إلى موارد أو تنفيذ عمليات خارج نطاق صلاحياته المقررة. يعني ذلك أن آليات التحقق من الصلاحيات (Authorization) مفقودة أو معطوبة أو سهلة التجاوز.

OWASP A01:2025: تحتفظ هذه الثغرة بالمرتبة الأولى في قائمة OWASP Top 10 للعام 2025 للسنة الثانية على التوالي. تشمل الآن أيضاً SSRF (Server-Side Request Forgery) ضمن نطاقها — وهو تعديل أضاف 5 CWEs جديدة تجعل المجموع 40 CWE.

WR-01 · IDOR
PayPal — IDOR to Add Secondary Users
Critical PayPal HackerOne
💰 $10,500
// السيناريو

الباحث وجد أن الـ endpoint الخاص بإضافة مستخدمين ثانويين في حسابات PayPal Business يقبل user ID لأي مستخدم بدون التحقق من ملكية الحساب، مما يتيح إضافة أي مستخدم كـ secondary user في حساب تجاري لا يملكه المهاجم.

// خطوات الاكتشاف — Discovery Steps
  1. Open two PayPal Business accounts (attacker: A, victim: B)
  2. Intercept the request to add secondary user on Account A using Burp Suite
  3. POST /businessmanage/users/api/v1/users → body contains {"userId": "A_OWNED_USER_ID"}
  4. Replace userId with B's account user ID
  5. Send request — server returns 200 OK with user added to victim's account
  6. Verify: Log into account B, check secondary users list — attacker's email appears
🇸🇦 الخطوات بالعربي — اضغط للعرض ▼
  1. افتح حسابين PayPal Business: الأول للمهاجم (A) والثاني للضحية (B)
  2. افتح Burp Suite والتقط الـ request الخاص بإضافة مستخدم ثانوي في الحساب A
  3. لاحظ أن الـ request يُرسل إلى /businessmanage/users/api/v1/users ويحتوي على userId في الـ body
  4. غيّر الـ userId في الـ body من ID الحساب A إلى ID الحساب B (الضحية)
  5. أرسل الـ request — الـ server سيرد بـ 200 OK ويُضيف المهاجم لحساب الضحية
  6. تحقق: ادخل لحساب B وافحص قائمة المستخدمين الثانويين — ستجد إيميل المهاجم فيها
// الـ Payload
HTTP Request 
POST /businessmanage/users/api/v1/users HTTP/1.1
Host: www.paypal.com
Authorization: Bearer ATTACKER_TOKEN
Content-Type: application/json

{
  "userId": "VICTIM_BUSINESS_ACCOUNT_ID", // ← CHANGED
  "role": "ADMIN"
}

// Response: 200 OK — User added to victim's business account
// الإصلاح
✅ FIX APPLIED

تحقق الـ server الآن من أن الـ authenticated user يملك الـ business account قبل السماح بإضافة users. تم التحقق من ملكية الحساب server-side عبر ربط الجلسة بـ business account ID.

WR-02 · IDOR via GraphQL
HackerOne — Delete Hacker Certifications via GraphQL
Critical HackerOne GraphQL
💰 $12,500
// السيناريو

الباحث اكتشف أن الـ GraphQL mutation الخاصة بحذف شهادات الهاكرز على HackerOne تقبل أي hackerCertificationId بدون التحقق من الملكية، مما يتيح لأي مستخدم حذف شهادات مستخدمين آخرين.

// خطوات الاكتشاف — Discovery Steps
  1. Create account A and B on HackerOne
  2. Add certifications to account B (get certification ID from response)
  3. Using account A, open Burp Suite and send GraphQL request
  4. Use CreateOrUpdateHackerCertification mutation with B's cert ID
  5. Delete/overwrite certification of user B while authenticated as A
  6. Confirm deletion by checking B's profile
🇸🇦 الخطوات بالعربي — اضغط للعرض ▼
  1. أنشئ حسابين على HackerOne: الحساب A (المهاجم) والحساب B (الضحية)
  2. من الحساب B أضف شهادات (certifications) واحتفظ بـ certification ID من الـ response
  3. افتح Burp Suite وأرسل GraphQL request باستخدام الحساب A
  4. استخدم mutation باسم CreateOrUpdateHackerCertification مع ID الشهادة التابعة للحساب B
  5. أضف _destroy: true في الـ mutation لحذف شهادة الضحية
  6. تحقق من الحذف بالدخول لبروفايل الحساب B
// الـ Payload
GraphQL Mutation 
mutation DeleteCertification {
  CreateOrUpdateHackerCertification(input: {
    id: "VICTIM_CERT_ID_HERE",  // ← IDOR
    _destroy: true
  }) {
    hackerProfile {
      id
      certifications { id name }
    }
  }
}
// الإصلاح

تم إضافة ownership check في GraphQL resolver — يتحقق من أن الـ authenticated user هو صاحب الـ certification ID قبل تنفيذ أي mutation.

WR-03 · IDOR via Phone Number
Uber — Access Any User Data via Phone Number
Critical Uber HackerOne
💰 $10,000
// السيناريو

باحث اسمه mongobug اكتشف أن Uber بدلاً من استخدام User ID، كان يستخدم رقم الهاتف كمعرّف في الـ API. نظراً لأن أرقام الهواتف يمكن الحصول عليها بسهولة، أصبح من الممكن الوصول لبيانات أي مستخدم Uber.

// خطوات الاكتشاف — Discovery Steps
  1. Create Uber account and intercept API requests with Burp
  2. Identify endpoint: GET /api/v2/user?phone=YOUR_PHONE
  3. Notice that phone number is used as the identifier
  4. Replace your phone with a known victim's phone number
  5. Server returns victim's full profile including PII
  6. Phone numbers are obtainable from Uber receipts, referral links
🇸🇦 الخطوات بالعربي — اضغط للعرض ▼
  1. أنشئ حساب Uber والتقط الـ API requests باستخدام Burp Suite
  2. حدد الـ endpoint: GET /api/v2/user?phone=YOUR_PHONE
  3. لاحظ أن رقم الهاتف يُستخدم كمعرّف للمستخدم في الـ API — وهو خطأ أمني
  4. استبدل رقمك برقم هاتف الضحية (يمكن الحصول عليه من إيصالات Uber أو روابط الإحالة)
  5. الـ server سيُرجع بيانات الضحية الكاملة: الاسم، الإيميل، تاريخ الرحلات
  6. أرقام الهواتف سهلة الحصول عليها من إيصالات Uber أو روابط الإحالة الترويجية
// الـ Payload

GET /api/v2/user?phone=+1-VICTIM-PHONE HTTP/1.1
Host: riders.uber.com
Authorization: Bearer ATTACKER_TOKEN

// Response: 200 OK
{
  "name": "Victim Name",
  "email": "victim@email.com",
  "phone": "+1-VICTIM-PHONE",
  "trip_history": [...]
}
// الإصلاح
✅ FIX APPLIED

تم تقييد الوصول بحيث يرتبط كل request بـ JWT token صالح يثبت ملكية الحساب. لم يعد يمكن الاستعلام بأرقام هواتف خارجية.

WR-04 · IDOR — Info Disclosure
GitLab — IDOR in External Status Check API
Medium GitLab HackerOne
💰 $610
// السيناريو

ثغرة IDOR في الـ API الخاص بـ "External Status Checks" في GitLab تُسرّب معلومات عن أي status check في الـ instance لأي مستخدم مصادق عليه، بدون اشتراط صلاحيات على المشروع المحدد.

// خطوات الاكتشاف — Discovery Steps
  1. Login to GitLab instance with low-privilege account
  2. Browse to any project you have access to, set up external status check
  3. Capture the API request: GET /api/v4/projects/:id/external_status_checks/:check_id
  4. Enumerate check_id parameter with incremental values
  5. Access status checks from projects you do NOT have permission to view
  6. Response reveals project metadata and external service URLs

Developer Checklist — قائمة المراجعة للمطورين
Ordered by Priority — P1 = Fix Immediately
كيفية استخدام هذه القائمة: P1 = يجب الإصلاح فوراً قبل النشر. P2 = إصلاح في Sprint الحالي. P3 = Sprint القادم. P4 = Backlog مع monitoring.
🔴 P1 — أعلى أولوية (Critical)
  • 🛑
    تطبيق Server-Side Authorization على كل endpoint
    لا تعتمد أبداً على client-side checks فقط. كل API call يجب التحقق server-side من صلاحية المستخدم.
  • 🛑
    Deny by Default
    جميع الـ endpoints محجوبة افتراضياً. الوصول يُمنح صراحةً فقط — لا شيء "مفتوح تلقائياً".
  • 🛑
    Object Ownership Verification (Anti-IDOR)
    قبل قراءة/تعديل/حذف أي resource: تحقق من أن resource.owner_id == session.user_id. لا استثناء.
  • 🛑
    RBAC/ABAC موحّد ومركزي
    استخدم Role-Based أو Attribute-Based Access Control مطبق في مكان واحد — لا قرارات authorization متفرقة في الكود.
  • 🛑
    إزالة جميع Legacy/Test Endpoints من Production
    أي endpoint "للاختبار فقط" يجب إزالته أو تأمينه قبل النشر. الـ Optus breach بسببها.
  • 🛑
    JWT Validation صارم
    لا تقبل alg=none. تحقق من signature. لا تثق بـ claims دون verification. استخدم مكتبات موثوقة فقط.
🟠 P2 — عالي الأولوية (High)
  • ⚠️
    استبدال Sequential IDs بـ UUIDs
    IDs التسلسلية (1, 2, 3) سهلة الـ enumeration. استخدم UUIDs v4 لجميع الـ objects الحساسة.
  • ⚠️
    HTTP Method Restrictions صارمة
    كل endpoint يقبل HTTP methods محددة فقط. GET-only endpoints ترفض POST/PUT/DELETE بـ 405.
  • ⚠️
    Tenant Isolation في Multi-Tenant Apps
    كل query يُضيف WHERE tenant_id = ?. لا resource يُرجَع دون التحقق من الـ tenant.
  • ⚠️
    منع Mass Assignment
    استخدم allowlist للـ fields المقبولة في JSON body. لا تمرر الـ request body مباشرة للـ ORM.
  • ⚠️
    Rate Limiting على Sensitive Endpoints
    حد طلبات 60/min على profile/data endpoints لمنع enumeration attacks.
  • ⚠️
    CORS Configuration صارمة
    لا Access-Control-Allow-Origin: * على APIs تحتوي بيانات حساسة. Allowlist فقط.
  • ⚠️
    تأمين GraphQL Resolvers
    كل resolver يتحقق من authorization قبل إرجاع البيانات. لا trust في arguments من client.
  • ⚠️
    Indirect Object References
    بدلاً من تمرير database ID مباشرة، استخدم mapping layer — المستخدم يرى reference token، الـ server يحوّله لـ internal ID.
🟡 P3 — متوسط الأولوية (Medium)
  • 📋

1. Anti-IDOR Pattern — Server-Side Ownership Check
Python — Flask/Django 
# ❌ VULNERABLE — No ownership check
@app.route('/api/tickets/<int:ticket_id>')
def get_ticket(ticket_id):
    ticket = Ticket.query.get(ticket_id)  # IDOR — any ID works!
    return jsonify(ticket.to_dict())

# ✅ SECURE — Ownership verified
@app.route('/api/tickets/<int:ticket_id>')
@login_required
def get_ticket(ticket_id):
    ticket = Ticket.query.filter_by(
        id=ticket_id,
        owner_id=current_user.id  # ← ownership enforced
    ).first_or_404()
    return jsonify(ticket.to_dict())
2. RBAC Middleware Pattern
Node.js — Express 
// ✅ Centralized RBAC middleware
const requireRole = (roles) => (req, res, next) => {
  const userRole = req.user?.role;
  if (!userRole || !roles.includes(userRole)) {
    return res.status(403).json({
      error: 'Forbidden — Insufficient permissions'
    });
  }
  next();
};

// Usage: apply to all admin routes
app.use('/api/admin', requireRole(['admin', 'superadmin']));
app.delete('/api/users/:id',
  requireRole(['admin']),
  async (req, res) => { /* safe — role checked */ }
);
3. Anti-Mass Assignment — Input Allowlisting
Python — Django REST Framework 
# ❌ VULNERABLE — All fields accepted
class UserSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = '__all__'  # isAdmin, role, verified all writable!

# ✅ SECURE — Explicit field allowlist
class UserUpdateSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = ['first_name', 'last_name', 'email']
        # role, isAdmin, verified NOT in list → cannot be set by user

    def update(self, instance, validated_data):
        # Extra safety: explicitly remove privileged fields
        validated_data.pop('role', None)
        validated_data.pop('is_admin', None)
        return super().update(instance, validated_data)
4. Secure JWT Validation

// ❌ VULNERABLE — Accepts alg=none

// ✅ SECURE — Strict algorithm + secret
const verifyToken = (token) => {

import ipaddress
from urllib.parse import urlparse

ALLOWED_SCHEMES = {'https'}
BLOCKED_HOSTS = {'localhost', '127.0.0.1', '0.0.0.0', '::1'}
BLOCKED_RANGES = [
    ipaddress.ip_network('10.0.0.0/8'),    # Private
    ipaddress.ip_network('172.16.0.0/12'),  # Private
    ipaddress.ip_network('192.168.0.0/16'), # Private
    ipaddress.ip_network('169.254.0.0/16'), # Link-local (AWS metadata)
]

def is_safe_url(url: str) -> bool:
    parsed = urlparse(url)
    if parsed.scheme not in ALLOWED_SCHEMES: return False
    if parsed.hostname in BLOCKED_HOSTS: return False
    try:
        ip = ipaddress.ip_address(parsed.hostname)
        if any(ip in r for r in BLOCKED_RANGES): return False
    except ValueError: pass  # hostname, not IP
    return True
8. Secure Direct Object Reference — Indirect Mapping
Python — Indirect Reference Map 
import secrets

# Per-user session mapping: public_token → internal_id
# Attacker cannot guess or enumerate internal IDs

class IndirectRefMap:
    def __init__(self):
        self._map = {}  # {public_token: internal_id}

    def add(self, internal_id: int) -> str:
        token = secrets.token_urlsafe(16)  # cryptographically random
        self._map[token] = internal_id
        return token

    def resolve(self, token: str) -> int:
        internal_id = self._map.get(token)
        if not internal_id:
            raise ValueError('Invalid reference')
        return internal_id

# No sequential IDs exposed to client at all
