Mutation XSS (mXSS) — تجاوز الـ Sanitizers mXSS Writeups 2025-2026 جديد ومتطور
ما هو الـ mXSS وليه خطير؟

الـ Mutation XSS هو إن الـ Sanitizer (زي DOMPurify) بيشوف الـ HTML ويحكم إنه آمن، لكن لما الـ browser يحط الـ HTML ده في الـ DOM، بـ"يصلح" بعض الـ tags بطريقة تخلي الـ payload الميت يتنفذ تاني. السبب هو إن في HTML 3 namespaces: HTML، SVG، و MathML — كل واحد بيتـ parse بطريقة مختلفة. الـ Sanitizer شغال في namespace معين، الـ browser بيـ render في namespace تاني.

CVE-2025-26791 — DOMPurify Bypass via Template Literal Regex Bug
DOMPurify before v3.2.4 · فبراير 2025 · Reported via GitHub Advisory

السيناريو: أي موقع بيستخدم DOMPurify مع خيار SAFE_FOR_TEMPLATES: true لتنظيف الـ HTML اللي اليوزر بيكتبه — زي التعليقات أو الـ rich text editor.

كيف اكتشفه الباحث:

خطوة 1: راجع الـ source code بتاع DOMPurify وركز على الـ Regex اللي بيتعامل مع Template Literals لما SAFE_FOR_TEMPLATES يكون enabled.
خطوة 2: لاحظ إن الـ Regex كان هو /\$\{[^{]+\}/g — ده الـ regex بيحاول يشيل الـ template literals زي ${...} لكن مش strict بما يكفي.
خطوة 3: جرب input بيبدو عادي للـ Regex لكن بيحتوي على SVG مع desc tag بطريقة معينة.
خطوة 4: اكتشف إن الـ browser لما بيـ parse الـ SVG بيـ"يصلح" الـ nodes بطريقة تخلي الـ malicious code يتنفذ بعد ما الـ sanitizer خلصه.
Payload:
DOMPurify.sanitize('<svg><desc>\onload=\${alert("XSS!")}\</desc></svg>', { USE_PROFILES: { svg: true }, SAFE_FOR_TEMPLATES: true });
// النتيجة: يعدي الـ sanitizer لكن المتصفح ينفذه

الإصلاح: DOMPurify شددوا الـ Regex في v3.2.4 وأضافوا فحوصات إضافية لـ SVG edge cases.

GHSA-H8R8-WCCR-V5F2 — mXSS via Re-Contextualization في DOMPurify
DOMPurify before v3.3.2 · مارس 2026 · أحدث bypass موثق

السيناريو: أي موقع بيستخدم DOMPurify بأي version قبل 3.3.2 لعرض HTML من اليوزر في الصفحة.

كيف اكتشفه الباحث:

خطوة 1: ركز على "إعادة التحويل للسياق" — إن الـ HTML المتكوّن في HTML namespace لما بيتنقل لـ MathML أو SVG namespace بيتـ parse تاني بشكل مختلف.
خطوة 2: استخدم الـ attribute is= في HTML5 اللي بيسمح لـ custom elements — ده بيخلق سياق جديد الـ sanitizer مش بيحسبه.
خطوة 3: بنى payload بيستخدم nested namespaces: HTML → MathML → SVG → HTML مع <textarea> لتغيير الـ parsing context.
Payload:
<math><annotation-xml encoding="text/html">
<x><svg><mtext><textarea>
<a is="</textarea><img src onerror=alert(origin)>">
</math>
// DOMPurify بيشوف textarea ويعتبره safe، المتصفح بيقفله ويشغل الـ img

الإصلاح: ترقية DOMPurify لـ v3.3.2+. الفريق أضاف فحص خاص للـ is= attribute والـ namespace transitions.

DOMPurify Bypass via XML Processing Instructions — @slonser_
Flatt Security Research · 2024 (مازال relevant في 2025) · تم الإصلاح في DOMPurify

السيناريو: التطبيق بيـ parse XML أو XHTML وبيستخدم DOMPurify لتنظيفه.

كيف اكتشفه الباحث:

خطوة 1: لاحظ إن HTML parser والـ XML parser بيتعاملوا مع الـ Processing Instructions (اللي بتبدأ بـ <?) بشكل مختلف.
خطوة 2: الـ HTML spec بيقول إن <? في HTML context بتتحول لـ comment وبتنتهي بأول >. لكن الـ XML spec بيقول إنها بتنتهي بـ ?>.
خطوة 3: DOMPurify كان بيستخدم NodeFilter.SHOW_ELEMENT | SHOW_COMMENT | SHOW_TEXT فقط — مش بيشوف الـ Processing Instructions أصلاً.
خطوة 4: عمل payload بيستغل الـ mismatch ده — الـ HTML parser بيشوف comment، DOMPurify ما بيفحصوش، الـ browser بعدين بيـ render الـ img tag.
Payload:
<?xml-stylesheet >
<img src=x onerror="alert('DOMPurify bypassed!!!')">
?>

الإصلاح: DOMPurify أضاف NodeFilter.SHOW_PROCESSING_INSTRUCTION عشان يفحص الـ Processing Instructions ويشيلها.

Prototype Pollution → XSS Client-Side Prototype Pollution Writeups لسه active في 2025
الفكرة ببساطة

في JavaScript، كل object بيرث من Object.prototype. لو مهاجم قدر يضيف property فيه — مثلاً Object.prototype.innerHTML = "XSS" — وفي الكود "script gadget" بيستخدم property بدون فحص وجودها، ده بيؤدي لـ XSS حتى من غير أي input field تقليدي. يحصل عادة عبر URL parameters زي ?__proto__[x]=y أو عبر JSON merge functions.

Prototype Pollution → XSS via jQuery deparam + Google Analytics Gadget
Intigriti XSS Challenge · موثق ومُستخدم في real-world programs

السيناريو: الموقع بيستخدم مكتبة jquery-deparam لقراءة الـ URL parameters وتحويلها لـ objects.

خطوات الاكتشاف:

خطوة 1: فتح الموقع وعمل نظرة على الـ JS files — وجد jquery-deparam.js و analytics.js من Google.
خطوة 2: بحث عن CVE لـ jquery-deparam — وجد إنها vulnerable لـ Prototype Pollution عبر URL parameters زي ?__proto__[test]=value.
خطوة 3: جرب في الـ developer console بعد زيارة الـ URL: console.log(test) — ظهرت القيمة! يعني الـ Prototype Pollution شغال.
خطوة 4: دور على "script gadget" — كود في نفس الصفحة بيستخدم property من الـ prototype. وجد إن Google Analytics بيستخدم cookieName property لتحديد اسم الـ cookie.
خطوة 5: بنى payload يلوث Object.prototype.cookieName بـ XSS payload. Google Analytics بيكتب الـ cookie باستخدام document.cookie = cookieName + "=" + value.
Final URL:
https://target.com/?__proto__[cookieName]=username%3D%3Cimg%20src%3Dhttps%3A%2F%2Fattacker.com%2Fx.jpg%20onload%3D%22alert(document.domain)%22%3E%3Cstyle

الإصلاح: تحديث jquery-deparam، أو استخدام Object.freeze(Object.prototype)، أو إزالة المكتبة والاعتماد على URLSearchParams المدمجة.

DOM XSS via postMessage Cross-Window Messaging Abuse Writeups متكرر في Modern Apps
الفكرة ببساطة

الـ postMessage API بتسمح للـ windows والـ iframes يتواصلوا مع بعض. المشكلة لما الكود بيـ listen لـ messages ومش بيتحقق من event.origin — أي صفحة تانية تقدر تبعت message وتتحكم في الـ DOM. الأخطر إن الـ third-party widgets زي Marketo forms، payment SDKs، وanalytics libraries كتير منها بتستخدم postMessage ومش دايماً بتعمل origin validation صح.

DOM XSS في HackerOne.com نفسها via Marketo postMessage
HackerOne Report #398054 · $500 Bounty · 204 upvotes · أحد أشهر الـ reports

السيناريو: موقع hackerone.com كان بيستخدم Marketo contact form widget — ده widget بيتحط كـ third-party JavaScript.

خطوات الاكتشاف:

خطوة 1: فتح صفحة الـ contact في hackerone.com وعمل inspect للـ page source. لاحظ Marketo form script محمل.
خطوة 2: فتح الـ JS console وبحث عن addEventListener('message' في الـ scripts — وجد listener فيه.
خطوة 3: راجع الكود — الـ listener بيستقبل الـ message ولو فيه mktoResponse بيستدعي findCorrectFollowUpUrl() بعدين بيعمل window.location = url.
خطوة 4: مفيش فحص لـ event.origin — أي صفحة تقدر تبعت message.
خطوة 5: بنى صفحة exploit تفتح hackerone.com في popup ثم تبعتله message بـ javascript: URL:
Exploit Code:
var w = window.open('https://hackerone.com/contact');
setTimeout(function() {
w.postMessage(JSON.stringify({
mktoResponse: { followUpUrl: 'javascript:alert(document.domain)' }
}), '*');
}, 2000);

الإصلاح الأول كان خاطئ: HackerOne أضافوا فحص بـ indexOf — لكن indexOf مش safe! إذا المهاجم سجّل domain اسمه app-sj17.ma (نفس بداية الـ Marketo domain) كان يقدر يـ bypass.

الإصلاح الصح:
// خطأ — indexOf قابل للـ bypass
if (trustedOrigin.indexOf(event.origin) === 0) { ... }

// صح — strict equality
if (event.origin === 'https://app-sj17.marketo.com') { ... }

Blind XSS — الثغرة اللي بتتأخر Out-of-Band XSS Writeups 33% من XSS في Bug Bounty 2025
ليه Blind XSS بقى الأعلى أجراً في 2025؟

الـ Blind XSS بيشتغل لما الـ payload بيتخزن في جهة (زي feedback form أو user profile) لكن بيتنفذ في جهة تانية (زي admin dashboard أو support system) من غير ما المهاجم يشوف الـ execution مباشرة. ده خطير لأنه بيتنفذ في high-privilege context — يعني ممكن يوصل لبيانات الـ admin أو لـ internal systems مش متاحة من الخارج. في 2025، Bugcrowd وثّق bounties تجاوزت $250,000 من Blind XSS في admin panels.

Self-XSS → XSSJacking → Real XSS على Victims

السيناريو: تطبيق فيه "Self-XSS" — يعني XSS بيشتغل بس على حساب المهاجم نفسه، مش على حساب الضحية. عادة البرامج ما بتقبلش تقارير Self-XSS لأن التأثير محدود.

كيف الباحث حوّله لـ Real XSS:

خطوة 1: وجد Self-XSS في حقل review email — الـ payload بيشتغل بس لما المهاجم نفسه يفتح الصفحة.
خطوة 2: لاحظ إن الصفحة ممكن تتحط في iframe (مفيش X-Frame-Options protection).
خطوة 3: بنى attack page بتستخدم XSSJacking — بتحط الـ vulnerable page في iframe غير مرئي وبتخلي الضحية تعمل actions معينة (زي paste) بدون ما يعرف.
خطوة 4: الضحية لما يزور الـ attack page، بتستخدم social engineering تخليه يعمل Ctrl+V في نافذة مخفية — الـ clipboard بيحتوي على الـ XSS payload اللي المهاجم جهّزه.
خطوة 5: الـ XSS بيتنفذ في context الضحية وليس المهاجم — بقى Real XSS.

الإصلاح: X-Frame-Options: DENY + CSP: frame-ancestors 'none' + تصليح الـ Self-XSS نفسه.

الحلول الكاملة — Checklist للـ Dev Team Defense Strategies — Pre-Production Security
✅ Checklist الكامل قبل الرفع للـ Production
1
Output Encoding حسب الـ Context: مش encoding واحد يكفي — HTML context: HTML entities، JS context: \uXXXX، URL context: percent encoding، CSS context: \XX. كل context محتاج encoding مختلف.
2
DOMPurify >= 3.3.2: أي version أقل من ده vulnerable لـ mXSS. فحص كل package.json وnpm audit — ده أهم حاجة لو بتستخدم rich text.
3
Content Security Policy صارم: script-src 'self' 'nonce-RANDOM_PER_REQUEST' + فحص كل trusted domain للـ JSONP endpoints + X-Content-Type-Options: nosniff على كل response.
4
HttpOnly + SameSite=Strict على كل الـ Session Cookies: حتى لو الـ XSS نجح، الـ cookies مش متاحة لـ JavaScript ومش هتتبعت لـ external domain.
5
postMessage Origin Validation — Strict Equality:
if (event.origin !== 'https://exact-domain.com') return;
مش indexOf ولا includes ولا startsWith.
6
Object.freeze(Object.prototype): أول سطر في أول script في كل صفحة — بيمنع Prototype Pollution. أو استخدام structuredClone() بدل الـ deep merge.
7
X-Frame-Options: DENY + CSP frame-ancestors 'none': بيمنع XSSJacking وClickjacking على صفحات حساسة.
8
Trusted Types API: Browser API بتمنع الـ DOM sinks (innerHTML, document.write) من قبول raw strings — بتفرض المرور عبر Trusted Types policy.
9
Subresource Integrity (SRI) على كل 3rd-party scripts:
<script src="https://cdn.../x.js" integrity="sha384-HASH" crossorigin="anonymous"></script>
10
فحص الـ 3rd-party libraries بـ npm audit / Snyk: Mermaid, Chart.js, rich text editors, jQuery plugins — كلها ممكن تكون vulnerable. اعمل audit قبل كل deploy.
11
LLM Output Sanitization: لو الموقع فيه AI features — عامل الـ AI output عمل user input وطبق نفس الـ sanitization قبل عرضه في الـ DOM.
12
Paste Events Sanitization في الـ Rich Text Editors: الـ paste event ممكن يحقن HTML من الـ clipboard — لازم يتـ sanitize بنفس طريقة الـ direct input.
CSP Header المثالي
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-{RANDOM_PER_REQUEST}'; style-src 'self' 'nonce-{RANDOM_PER_REQUEST}'; img-src 'self' data: https:; connect-src 'self' https://your-api.com; frame-ancestors 'none'; base-uri 'self'; form-action 'self'; object-src 'none'; require-trusted-types-for 'script';
⚠️ تحذير: لو محتاج تـ whitelist أي CDN (cdnjs, googleapis, github.io) — افحص كل الـ JSONP endpoints في الـ domain ده. Angular على cdnjs ممكن يـ bypass الـ CSP بالكامل. الحل الأفضل: استخدم Subresource Integrity بدل الـ domain whitelist.
